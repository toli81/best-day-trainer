APPROACH SELECTION
------------------------------------------------------------

Approach A (SELECTED): Dashboard-First
- Build dashboard shell and pages first using existing data
- Most pages work immediately (sessions, exercises, muscle groups, reps/sets all exist)
- Add form scoring as the last step, backfill existing exercises
- Pros: Visible progress fast, can course-correct UI before pipeline work
- Cons: Form page shows empty state temporarily during development

Approach B (Not selected): Pipeline-First
- Build form scoring first, reprocess sessions, then build UI on top
- All pages have data from day one but slower to see anything visual

DESIGN SECTIONS PRESENTED & APPROVED
------------------------------------------------------------

Section 1: Layout Architecture
- Next.js route groups: (dashboard) for new layout, (legacy) for existing pages
- Root layout becomes minimal (html/body/ThemeProvider only)
- Dashboard pages at top-level routes: /, /volume, /form, /balance, /sessions, /notes
- Existing pages (/record, /upload, /library, /sessions/[id]) keep current header layout
- Both route groups share /sessions URL namespace intentionally
- Desktop: sticky top nav (logo, client selector, Record/Upload/Library links, theme toggle) + tab bar below
- Mobile: simplified top nav + fixed bottom tab bar (5 tabs + More)

Section 2: Dashboard Pages & Data
- Overview (/): 4 KPI stat cards, time range selector, combined area chart (volume + form), recent sessions
  - KPI cards: Total Sessions, Consistency %, Monthly Volume, Avg Form Score
  - Consistency for ranges < 2 weeks shows session count instead of percentage
- Volume (/volume): bar chart of reps over time, filter by muscle group/exercise
- Form (/form): line chart of form scores, AI score as solid line, override as distinct marker
- Balance (/balance): donut chart of muscle group distribution
- Sessions (/sessions): calendar heatmap + session list (absorbs current homepage)
- Notes (/notes): chronological feed of AI session notes
- Client selector in top nav: "All Clients" default + individual clients, persists via ?client= URL param
- Empty states for no-data scenarios

Section 3: API Routes
- 6 dashboard API routes, all accept ?client= and ?range= params
- GET /api/dashboard/stats — KPI card data
- GET /api/dashboard/volume — volume chart data (+ muscleGroup/exercise filters)
- GET /api/dashboard/form — form score chart data (+ exercise filter)
- GET /api/dashboard/balance — muscle group distribution
- GET /api/dashboard/sessions — heatmap + session list
- GET /api/dashboard/notes — notes feed
- PATCH /api/exercises/[exerciseId]/form-score — trainer override
- Effective score logic: formScoreOverride ?? formScore

Section 4: Form Scoring Pipeline Extension
- New pipeline stage: form_scored (between tags_generated and complete)
- Full stage order: downloaded -> uploaded_to_gemini -> overview_complete -> clips_extracted -> details_complete -> notes_generated -> tags_generated -> form_scored -> complete
- Single Claude API call per session with all exercises + scoring rubric (1-10 scale)
- Backfill endpoint: POST /api/admin/backfill-form-scores (synchronous, one session at a time)
- Trainer override UI: inline edit on session detail page

Section 5: Mobile Experience
- Bottom tab bar: 5 visible tabs (Overview, Volume, Form, Balance, Sessions) + More
- More sheet: Notes, Record, Upload, Library
- Simplified top nav: logo + client selector only
- KPI cards: 2x2 grid
- Charts: full-width, responsive via Recharts <ResponsiveContainer>
